One public file-search command for agents

파일 하나 찾을 때마다 38,650 토큰 쓰는 거 실화냐?

Jikji find는 파일 탐색 요청마다 raw agent가 쓰는 반복 LLM 호출, 토큰, 시간을 줄인다. 핵심은 0-call 홍보가 아니라, raw보다 낮아지지 않는 정확도와 훨씬 작은 탐색 예산이다.

Fullset value proof

HippoCamp Fullset 551 cases. 가격 추정은 입력 $0.30/1M, 출력 $2.50/1M, 환율 1,380 KRW/USD 기준이다. 모델 가격은 환경별로 달라질 수 있으므로 입력/출력 토큰을 별도로 보존한다.

절약과 정확도의 원인

Jikji find는 매번 파일시스템을 뒤지는 대신, prepare 때 만든 local retrieval context를 사용한다.

사전 파싱 캐시

문서/텍스트/미디어 본문을 미리 추출해 검색 시 원본 열기와 파싱 반복을 줄인다.

파일 지도와 route

폴더 맥락, 파일 카드, 중복 힌트, route row가 경로 탐색을 후보 선택으로 바꾼다.

LLM Wiki와 그래프

에이전트 친화적인 wiki source page와 knowledge graph route를 후보 검색에 사용한다.

여러 검색의 top-k 병합

메타데이터, 본문, 그래프, wiki/cache, 로컬 인덱스 후보를 dedupe한 slate로 만든다.

짧은 evidence 우선

agent는 큰 원본 대신 next_read와 evidence_pack을 먼저 읽어 토큰을 아낀다.

로컬 RAG-style

벡터DB/클라우드 RAG 필수 제품이 아니라, 로컬 wiki/graph/context를 쓰는 검색 레이어다.

에이전트 호출 규칙

Jikji find가 후보를 만들고, 에이전트는 그 후보 안에서 판단한다. raw 탐색은 JSON 계약이 허용할 때만 fallback이다.

jikji find ROOT "query" --json
# direct_use: answer_paths를 그대로 사용
# jikji_retry: 더 날카로운 find 재시도 1회
# raw_fallback_after_retry: 재시도 실패 뒤에만 raw 탐색 허용
Detailed benchmark ledger: jikji-benchmarks.html#usage-ledger.
